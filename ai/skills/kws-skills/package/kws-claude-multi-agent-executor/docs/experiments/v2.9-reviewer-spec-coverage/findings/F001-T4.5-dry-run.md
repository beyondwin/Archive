## Critical finding — failure mode shifted, not eliminated

**F002 baseline (v2.6.0)**: Reviewer did not consider `30m20m` at all. The
case was simply absent from review output. 3 of 4 reps shipped buggy code
with `SPEC_STATUS: PASS`.

**v2.9 T4.5 rep 1**: Reviewer explicitly considered `30m20m`. Generated it
as an adversarial input. Reasoned about whether it should be rejected. And
concluded:

> *"the spec grammar `<integer><unit>` concatenated allows repeated units;
> the spec example `30m1h` shows order doesn't matter; the spec does not
> prohibit repeated units; accepted: returns 3000; this is not a bug."*

The Reviewer's reasoning is **defensible against the spec text itself**.
The fixture's spec excerpt:
- Lists 5 explicit ValueError examples (empty / bare number / unknown unit /
  negative / decimal). `30m20m` is not among them.
- Includes a meta-rule "strict validation" but does not enumerate "repeated
  units" as a violation.
- Includes `30m1h` as a happy-path showing order doesn't matter, which a
  reasonable reader might interpret as "the grammar permits any concatenation
  of `<integer><unit>` segments, including repeated units."

The fixture's *rubric* encodes "repeated unit → ValueError" as authoritative
truth, but the spec text used as Reviewer input is ambiguous on this case.
F002 framed the miss as Sonnet's judgment ("'have I seen this unit?' judgment
call"), and v2.9's pilot result *makes that judgment explicit instead of
silent*.

## Implications for T5 decision

The walk mechanism does what v2.9 designed it to do — it inverts the
question, surfaces adversarial cases, and forces the Reviewer to reason
about them. The mechanism is working.

The rubric outcome on fixture 08 has not moved (0.95 in this rep, same as
F002 baseline). But the cause has shifted from *missing* (F002) to
*spec-text ambiguity surfaced and reasoned through* (v2.9).

Two paths for what to do next:

### Path α — Ship v2.9 as drafted, accept the new failure mode

Justification: the walk mechanism is empirically firing and producing
explicit reasoning. The current `30m20m` outcome is now a spec-vs-rubric
disagreement, not a Reviewer-quality failure. To move the rubric outcome,
the *fixture's spec excerpt* needs to be clarified (explicitly forbid
repeated units), or the *fixture's rubric* needs to be revised (accept
`30m20m` as valid given spec text).

Trade-off: declares victory on a Reviewer-quality improvement that is
real (no more silent miss) but invisible in the existing rubric metric.

### Path β — Iterate sub-step B with "prefer-strict on ambiguity" directive

Add to sub-step B: *"When a meta-rule is ambiguous and an adversarial
input could plausibly violate strict validation, mark NOT FOUND or PARTIAL
rather than satisfied — i.e., when in doubt, prefer the stricter
interpretation."*

This may push the rubric outcome but risks false-positive flags on
legitimately ambiguous cases in other fixtures. Single-fixture
optimization risk.

Cost to test: 1 more rep ($5-10).

### Path γ — Run T5 (n=3-4) with the current prompt before deciding

Justification: n=1 is too small. Re-running with the current prompt 3-4
times reveals whether the spec-reasoning outcome is reproducible across
reps. If 4/4 reps reason through `30m20m` explicitly (regardless of the
verdict reached), the walk mechanism is solidly working.

Cost: $20-40.

## Recommendation

Path γ then α. T5 (n=3-4) on the current prompt confirms the walk
mechanism is reproducible. If yes, ship v2.9 with the finding that the
intervention solved the silent-miss failure mode and surfaced a different
underlying issue (fixture spec ambiguity) that requires a separate fix
(fixture spec clarification, recorded as v2.9.x candidate).

Do not pursue Path β (single-fixture prompt optimization). The risk of
over-tuning sub-step B's directive language on one fixture's specific
ambiguity outweighs the marginal gain in rubric outcome.

작업 원칙:
- 편집 전에 repo-local instructions를 확인해. 예: `AGENTS.md`, `CLAUDE.md`, task-specific agent guide.
- 문서 계획과 repo instructions가 충돌하면 repo instructions를 우선하고, 계획 의도에 맞는 최소 수정으로 적응해.
- subagent-driven-development 방식으로 계획 문서의 첫 Task(Task 0 또는 Task 1)부터 마지막 Task까지 순서대로 수행해. 기본 목표는 현재 세션에서 가능한 한 끝까지 완료하는 것이다.
- 각 Task 시작 전에 본문 5줄 이내 `TASK EXECUTION CONTRACT`를 확정해: `scope`, `files_to_inspect`, `allowed_edits`, `forbidden_edits`, `acceptance_command_or_honest_substitute`. 계획 문서만으로 채울 수 없으면 repo를 확인해 보강하고, 실행 범위를 바꿀 수 있는 공백이면 blocker로 보고해.
- 계획의 Task가 여러 레이어나 모듈을 동시에 넓게 건드리면 문서 의도는 유지하되 Task 내부 Phase를 한 레이어/모듈 중심으로 나누어 진행해. 각 Phase도 필요한 파일, 허용/금지 수정 범위, 검증 기준을 자기완결적으로 명시해.
- 각 Task는 fresh implementation subagent로 구현해. 구현 subagent의 자기 보고만으로 review를 대체하지 말고, `gpt-5.5 high`가 spec compliance review와 code quality review를 수행해. 고위험/큰 Task는 별도 fresh reviewer subagent를 둘 수 있다. 필요한 수정/재리뷰, 검증, agent cleanup까지 끝난 뒤 Task를 닫아. 큰 Task를 Phase로 나누거나 continuation handoff가 발생해도 재개 Phase는 fresh implementation subagent로 진행해.
- 기본 판단 라우팅은 `gpt-5.5` + reasoning effort `high`다. 구현 방향, 파일 수정, 리뷰, root-cause, 결과 해석, 검증 해석, architecture/state/auth/persistence/shared-module 판단, 완료 판단은 모두 `gpt-5.5 high`로 수행해.
- 보수적 자동 Spark 예외: 사용자가 Spark/model optimization을 금지하거나 `gpt-5.5 only`를 요구하지 않았다면, `gpt-5.5 high`가 명시적으로 선택한 read-only 명령/파일에 한해 `gpt-5.3-codex-spark` + reasoning effort `high`를 evidence packing 용도로 자동 사용할 수 있다.
- Spark evidence packing은 non-mutating verification command 실행, raw stdout/stderr 파일 보존, exit code/cwd/command/duration 기록, failing test name/error excerpt/stack trace anchor 추출, `git status`/`git diff --stat`/`git diff --name-only` 같은 read-only 상태 요약, `HANDOFF CHECKPOINT`의 `verification`/`raw_output` 필드 초안에만 허용된다.
- Spark는 관련 소스 파일을 스스로 고르거나 탐색 범위를 넓히지 않는다. 구현 방향, root-cause 가설/결론, 리뷰 코멘트, 검증 해석, 완료 판단, 파일 편집, formatter/dependency install/migration/cleanup/service lifecycle, staging/commit/merge/push/PR/release/version/changelog 결정은 금지한다.
- Spark 출력은 증거 패킷일 뿐이다. root-cause, 수정, 리뷰, 검증 해석, 완료 판단 전에는 `gpt-5.5 high`가 핵심 파일과 raw output 경로 또는 핵심 excerpt를 직접 확인해.
- 위 보수적 Spark evidence packing과 사용자가 명시한 모델별 예외 외에는 다른 모델로 Task나 지원 작업을 라우팅하지 마.

- checkpoint는 마지막 checkpoint 이후 delta 중심으로 헤더 제외 10줄 이내의 고정 필드로 작성해: `status: completed | blocked | error`, `completed_scope`, `changed_files`, `contracts_confirmed`, `verification`, `raw_output`, `remaining_risk`, `next_action`, `worktree_branch`, `session_owned_resources`.
- summary나 checkpoint는 재개에 필요한 사실만 남겨: 생성/수정 파일, 확정된 interface/contract, 검증 결과, 다음 액션. 감상, 일반 설명, 긴 로그 전문은 넣지 마.
- 자발적으로 `CONTINUATION PROMPT`를 출력하고 멈추는 것은 예외다. 사용자 결정 필요, unresolved root cause/verification failure, 다음 고위험 작업을 안정적으로 재구성할 수 없음, 실제 context 압박이 명확한 경우에만 멈춰.
- mid-task, mid-review, failed-verification 분석 중에는 handoff하지 말고 먼저 원인과 상태를 정리해 semantic boundary를 만들어라. Task 개수, 영역 전환, 변경 파일 증가, 긴 로그 누적만으로는 멈추지 마.

- 수정 후 같은 검증 또는 더 작은 재현 검증을 다시 실행해.
- 같은 root cause에 대한 수정 재시도는 최대 3회로 제한해. 3회 후에도 실패하면 `status: error` 또는 `status: blocked` checkpoint로 시도한 수정, raw output 위치, 남은 가설, 다음 검증을 정리하고 멈춰.

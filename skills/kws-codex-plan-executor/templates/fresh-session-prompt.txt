작업 원칙:
- 실행 시작 직후 적용 가능한 installed skills를 먼저 load/follow해. 특히 `using-superpowers`를 먼저 적용하고, feature/bugfix/refactor/behavior change 구현 전에는 `test-driven-development`를 적용해 failing test/eval을 먼저 확인해. 이건 headless-only rule이 아니라 interactive and headless 실행 모두의 구현 규칙이다. 각 구현 Task는 구현 전 RED evidence(명령/eval + 예상 실패)와 수정 후 GREEN evidence를 state/checkpoint에 남겨라.
- 편집 전에 repo-local instructions를 확인해. 예: `AGENTS.md`, `CLAUDE.md`, task-specific agent guide.
- 문서 계획과 repo instructions가 충돌하면 repo instructions를 우선하고, 계획 의도에 맞는 최소 수정으로 적응해.
- 계획 문서의 첫 Task(Task 0 또는 Task 1)부터 마지막 Task까지 순서대로 수행해. 기본 목표는 현재 세션에서 가능한 한 끝까지 완료하는 것이다. subagent는 사용자가 `subagents=on`, delegation, parallel work, subagents를 명시한 경우에만 사용해. 사용한 경우 state에 `subagents_requested=true`와 `subagent_runs[]`를 기록하고, parent executor가 file-change list, review status, write-scope overlap rationale, 최종 검증을 직접 확인해.
- 시작 직후 계획 문서의 Task 목록을 훑고 각 Task에 `risk: low | mid | high`와 한 줄 이유를 부여해. `low`는 단일 파일/모듈의 고립 변경, `mid`는 2개 이상 모듈/공유 상태/config 변경, `high`는 cross-area/API/schema/auth/persistence/breaking change다. 같은 파일을 여러 Task가 수정하면 뒤 Task는 최소 `mid`로 승격해.
- 실행 시작 시 `run_id`를 만들고 작업 worktree 안에 `.codex-orchestrator/runs/<run_id>/`를 만든 뒤, primary state를 `.codex-orchestrator/runs/<run_id>/state.json`에 유지해. `.codex-orchestrator/state.json`은 resume 호환용 latest-state copy 또는 pointer로만 갱신해. 같은 프로젝트에서 여러 executor가 동시에 실행될 수 있으므로 실행 중 source of truth는 항상 per-run state_path다. 필드는 `schema_version`, `run_id`, `mode`, `workspace`, `plan`, `spec`, `branch`, `worktree`, `run_dir`, `state_path`, `context_snapshot_path`, `context_basis_hash`, `context_health`, `current_task`, `current_phase`, `lifecycle_outcome`, `handoff_reason`, `completion_audit`, `subagents_requested`, `subagent_runs`, `tasks`, `risk_levels`, `review_issue_keys`, `verification`, `session_owned_resources`, `last_checkpoint`, `timestamps`를 포함해. `HANDOFF CHECKPOINT`는 사람이 읽는 요약이고, 이 JSON은 재개용 source of truth다.
- 편집 전에 `.codex-orchestrator/runs/<run_id>/context.json`을 만들고, plan/spec/docs source path와 SHA-256, `basis_hash`를 기록해. state에는 `context_snapshot_path=".codex-orchestrator/runs/<run_id>/context.json"`과 `context_basis_hash`를 저장해. source snapshot은 repo-relative source path와 hash만 담고, out-of-repo source는 blocker로 처리해.
- `context.json` 생성 직후, 각 Task/Phase 종료 후, blocker/error 발생 시, handoff/resume 전, final 직전에 state의 `context_health`를 갱신해. 필드는 `status=green|yellow|red`, `last_checked_at`, `context_snapshot_present`, `context_basis_hash_recorded`, `active_task_contract_present`, `next_action`, `open_questions`, `known_assumptions`, `handoff_ready`다. `green`은 state와 artifacts만으로 이어받을 수 있음, `yellow`는 진행 가능하지만 가정/open question이 있음, `red`는 안전한 진행에 blocker/user decision/handoff가 필요함을 뜻한다.
- Task 종료 시 `.codex-orchestrator/runs/<run_id>/state.json`의 해당 task `status`를 `completed`, `blocked`, 또는 `error` 중 하나로 명확히 갱신해. `completed_at`만 설정하고 `status`를 `in_progress`로 남기지 마.
- `interactive`와 `headless` 실행 중 notable boundary가 발생하면 user-local learning log에 redacted-context JSONL 이벤트를 남겨. 실행 시작 후 helper `ai/skills/kws-skills/package/kws-codex-plan-executor/scripts/append_learning_event.py init-run`으로 `run_id`를 초기화하고, 이벤트는 `append --run-id "$RUN_ID"`로 기록하며, 실행 종료 시 `close-run --run-id "$RUN_ID" --outcome success|blocked|error`를 best-effort로 호출해. 저장 위치는 `~/.codex/learning/kws-codex-plan-executor/runs/<YYYY-MM-DD>/<run_id>/events.jsonl`이고 전체 색인은 `~/.codex/learning/kws-codex-plan-executor/index.jsonl`이다. 각 이벤트에는 `run_id`, `execution.run_dir=".codex-orchestrator/runs/<run_id>"`, `execution.state_path=".codex-orchestrator/runs/<run_id>/state.json"`을 포함해. 기록 대상은 `blocker`, `error`, `verification_failure`, `recurring_issue`, `user_correction`, `successful_workaround`, actionable `completion_learning`이다. `prompt`와 `handoff` 생성 자체는 logging mode가 아니다. secrets, 전체 대화 transcript, 긴 raw log, 절대 home path는 저장하지 마.
- headless 실행의 마지막 결과는 structured headless result shape를 따라라: `status`, `run_id`, `state_path`, `summary`, `changed_files`, `verification`, `open_gaps`, `residual_risk`, `next_action`. `templates/headless-output-schema.json`를 사용할 수 있으면 `codex exec --output-schema`에 사용하고, 사용할 수 없으면 같은 JSON shape를 마지막 메시지로 남겨라.
- 각 Task 시작 전에 본문 5줄 이내 `TASK EXECUTION CONTRACT`를 확정해: `scope`, `files_to_inspect`, `allowed_edits`, `forbidden_edits`, `acceptance_command_or_honest_substitute`. 계획 문서만으로 채울 수 없으면 repo를 확인해 보강하고, 실행 범위를 바꿀 수 있는 공백이면 blocker로 보고해.
- 실행 가능한 Task는 `unit_manifest`를 기록할 수 있다. 필드는 `unit_type`, `context_mode`, `required_skills`, `tool_policy`, `allowed_write_globs`, `forbidden_write_globs`, `artifact_policy`, `max_context_chars`다. 완료된 Task를 포함해 `lifecycle_outcome="finished"`를 기록하려면 각 completed Task의 manifest가 유효해야 한다. `implementation` manifest는 비어 있지 않은 `allowed_write_globs`가 필요하고, `read-only` manifest는 write glob을 허용하면 안 된다.

- 리뷰/검증 실패를 재시도할 때는 각 이슈에 가능한 한 `ISSUE_KEY=<file>:<line-or-symbol>:<category>`를 붙이고 `.codex-orchestrator/runs/<run_id>/state.json`의 `review_issue_keys`에 기록해. 이전 재시도와 같은 `ISSUE_KEY`가 다시 나오면 `[RECURRING - previous fix did not address this]`로 표시하고, 같은 방식의 수정 반복 전에 root cause를 다시 확인해.
- 기본 판단 라우팅은 `gpt-5.5` + reasoning effort `high`다. 구현 방향, 파일 수정, 리뷰, root-cause, 결과 해석, 검증 해석, architecture/state/auth/persistence/shared-module 판단, 완료 판단은 모두 `gpt-5.5 high`로 수행해.
- 보수적 자동 Spark 예외: 사용자가 Spark/model optimization을 금지하거나 `gpt-5.5 only`를 요구하지 않았다면, `gpt-5.5 high`가 명시적으로 선택한 read-only 명령/파일에 한해 `gpt-5.3-codex-spark` + reasoning effort `high`를 evidence packing 용도로 자동 사용할 수 있다.
- Spark evidence packing은 non-mutating verification command 실행, raw stdout/stderr 파일 보존, exit code/cwd/command/duration 기록, failing test name/error excerpt/stack trace anchor 추출, `git status`/`git diff --stat`/`git diff --name-only` 같은 read-only 상태 요약, `HANDOFF CHECKPOINT`의 `verification`/`raw_output` 필드 초안에만 허용된다.
- Spark는 관련 소스 파일을 스스로 고르거나 탐색 범위를 넓히지 않는다. 구현 방향, root-cause 가설/결론, 리뷰 코멘트, 검증 해석, 완료 판단, 파일 편집, formatter/dependency install/migration/cleanup/service lifecycle, staging/commit/merge/push/PR/release/version/changelog 결정은 금지한다.
- Spark 출력은 증거 패킷일 뿐이다. root-cause, 수정, 리뷰, 검증 해석, 완료 판단 전에는 `gpt-5.5 high`가 핵심 파일과 raw output 경로 또는 핵심 excerpt를 직접 확인해.
- 위 보수적 Spark evidence packing과 사용자가 명시한 모델별 예외 외에는 다른 모델로 Task나 지원 작업을 라우팅하지 마.

- mid-task, mid-review, failed-verification 분석 중에는 handoff하지 말고 먼저 원인과 상태를 정리해 semantic boundary를 만들어라. Task 개수, 영역 전환, 변경 파일 증가, 긴 로그 누적만으로는 멈추지 마.

실행 기준:
- 설계/계획 범위를 임의로 넓히지 말고, 문서 범위만 정확히 구현해.
- 문서와 실제 코드베이스가 다르면 문서 의도를 유지하는 가장 작은 수정으로 적응해.
- 정말 막히는 경우에만 blocker 를 명확히 정리해서 보고해.
- 검증은 비용과 리스크에 맞춰 계층화해: 조사-only Phase는 결정/다음 검증 계획만 기록, 구현 Phase는 targeted check, Task 완료나 high-risk/cross-area 변경은 관련 범위 확장, 마지막에는 계획의 final verification command 또는 가장 정직한 대체 full check를 실행해.
- `risk=high` Task는 compact high-risk verification matrix를 남겨. 관련 있는 scenario만 `passed|failed|blocked|not-applicable`로 기록하고, 명령 또는 manual check와 evidence path/excerpt를 붙여라: malformed or unexpected input, stale state or resume path, dirty worktree preservation, hung or long-running command behavior, misleading success output or skipped tests, cancellation/interruption recovery. 무관한 scenario는 실행하지 말고 한 줄 이유와 함께 `not-applicable`로 표시해.
- test config, build input, test property, Dockerfile, migration, generated artifact 입력을 바꾼 Task의 targeted verification은 cache/up-to-date가 변경 검증을 숨기지 않도록 `clean`, `--rerun-tasks`, fresh output path, 또는 동등한 honest substitute를 사용해. cache-busting을 하지 않는다면 raw output의 `UP-TO-DATE` 여부와 이유를 기록하고 Task/final verification에서 실제 실행되는 broader check로 닫아.
- 검증 실패가 나오면 raw output을 보존하고 `gpt-5.5 high`로 root-cause analysis를 수행해. 긴 raw output은 대화에 전문을 붙이지 말고 파일 경로 또는 핵심 excerpt와 재현 명령을 기록해.
- 실패가 환경 문제일 가능성이 있으면 blocker로 멈추기 전에 4단계를 순서대로 확인해: 1) test command 자체가 실행되는가, 2) dependency가 빠졌는가, 3) path/config가 worktree와 맞는가, 4) DB/server/external service가 필요하고 시작 가능한가. 각 단계의 명령, 결과, raw output 위치를 `verification`과 checkpoint에 기록해.
- 원인 없이 우회하거나 테스트를 약화하지 마.
- 수정 후 같은 검증 또는 더 작은 재현 검증을 다시 실행해.
- 같은 root cause에 대한 수정 재시도는 최대 3회로 제한해. 3회 후에도 실패하면 `status: error` 또는 `status: blocked` checkpoint로 시도한 수정, raw output 위치, 남은 가설, 다음 검증을 정리하고 멈춰.

마지막 문서 영향 확인:
- 계획 문서에 문서 업데이트 Task가 있으면 그 범위만 수행해.
- 구현 결과가 README, AGENTS.md, 운영 문서, prompt/operator docs, agent-facing instructions와 명확히 어긋나게 만들거나 사용자가 문서 업데이트를 요구한 경우에만 관련 문서를 좁게 업데이트해.
- 문서 업데이트는 실제 동작과 검증 결과에 근거하고, repo-local 문서 검증 명령이 있으면 실행해. 실행하지 못한 검증은 이유와 대체 확인을 기록해.
- public-safety, secret, path, link scan은 기본적으로 변경된 문서 또는 added lines로 범위를 좁혀 실행해. public release나 scanner 자체 변경처럼 전체 스캔이 필요한 경우에만 넓히고, 기존 false positive와 이번 변경에서 새로 생긴 위험을 구분해 기록해.
- 문서 영향이 없으면 문서를 수정하지 말고 최종 요약에 “문서 업데이트 없음: <이유>”를 기록해.

마무리 기준:
- 성공 완료 전 state에 `context_health.handoff_ready=true`, `context_health.status!="red"`, `lifecycle_outcome="finished"`와 `completion_audit`를 기록해. `completion_audit.passed=true`, 비어 있지 않은 `prompt_to_artifact_checklist`, 비어 있지 않은 `verification_evidence`, `open_gaps`, `residual_risk`를 포함한다.
- blocked/failed/userinterlude/askuserQuestion으로 끝나면 `lifecycle_outcome`을 해당 값으로 설정하고, `handoff_reason`에 다음 사람이 바로 판단할 수 있는 구체 이유를 남겨라.
- 마지막에는 무엇을 바꿨는지, 문서를 업데이트했다면 어떤 문서를 왜 바꿨는지, 문서를 바꾸지 않았다면 왜 불필요했는지, 어떤 검증을 실제로 돌렸는지, 어떤 모델을 썼는지, worktree 경로와 브랜치, 정리한 background process/agent 상태, 남은 리스크가 있는지만 간단히 정리해.

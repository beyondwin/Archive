## Capability map (what this version does)

Carries forward all v2.9.0 capabilities plus:

- **NEW v2.10.0**: 11th learning-log event type `context_health` —
  passive observation only. Emitted at:
  - Phase Transition T3 (after compaction completes and state is written)
  - Resume Chain chained-orchestrator startup (after `append-session-id`)
- **NEW v2.10.0**: Invariants table entry — `context_health` MUST NOT alter
  orchestrator control flow. Goodhart's-law guard documented in SKILL.md.
- Carried forward from v2.9.0: Reviewer Spec Coverage Walk (sub-step A
  + sub-step B), v2.8.1 enforced Step 7.5 init-run with `LEARNING_LOG_INIT:`
  marker, per-run sharded learning log, Resume Chain across compaction,
  Spec-edit branch (P15), review-side superpowers Skill calls, headless
  dispatch via `claude -p --dangerously-skip-permissions`.

### `context_health` required `context` fields (v2.10)

- `compaction_index` (int) — `state.last_compaction_after_task` derived index, or `-1` for chain-handoff snapshots
- `completed_tasks_count` (int) — count of `state.tasks[*].status == "COMPLETE"`
- `resume_chain_handoffs` (int) — `len(state.meta.session_ids) - 1`

### `context_health` optional `context` fields (v2.10)

- `risk_distribution` (object) — `LOW`/`MID`/`HIGH` → count
- `verifier_retry_total` (int) — sum across completed tasks
- `review_retry_total` (int) — sum across completed tasks
- `quality_trend_mean` (float) — mean of `state.quality_trend` if non-empty
- `drift_signals[]` (array of strings) — free-form observations

### `meta.json` / `events.jsonl` schema

Schema version unchanged (`"1"`). Existing field shape unchanged.
`skill_version` value bumped from `"2.9.0"` to `"2.10.0"` in helper.

### `state.json` schema

Documented in `../../ARCHITECTURE.md` §5. **No fields added in v2.10.0** —
`context_health` reads from existing state shape.

## Empirical state at ship

No empirical fixture re-run in v2.10.0 — schema-additive change with no
expected effect on existing fixtures. `context_health` is emitted but
not evaluated by any fixture rubric (it is observation infrastructure,
not a behavior under test).

Future v2.10.x or v2.11 work should:
- Capture ≥2 weeks of real-run `context_health` events.
- Decide which optional fields are load-bearing.
- Open `docs/experiments/v2.10-context-health/` if patterns suggest
  threshold-worthy metrics.

## Known limitations carried forward (status at ship)

All v2.9.0 limitations carry forward unchanged (see
`../risks-and-limitations.md`). New v2.10.0-specific limitations:

- **★ Insufficient `context_health` corpus**: v2.10.0 ships the emit
  infrastructure but has no real-run data yet. Aggregation/threshold
  decisions deferred until ≥2 weeks of runtime.
- **★★ Goodhart's-law risk (active management)**: keeping `context_health`
  passive is the only way to avoid threshold-driven behavior distortion
  before we know what the distributions look like. Documented as an
  invariant; future override requires explicit experiment.

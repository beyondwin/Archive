# v2.15 — Context Engineering (Specification)

**Skill version target**: `2.15.0` · **Schema version**: stays `"2"` (additive only) · **Backward compat**: full · **Hard dep**: v2.14.0 (`state.cost_ledger` field)

---

## Goals

| ID | Statement | Measurement |
|----|-----------|-------------|
| G1 | Per-Implementer-dispatch input token count drops materially when spec is large (>10 KB). | A/B vs v2.14 baseline on the same plan: median Implementer `input_tokens` ratio (post / pre) ≤ 0.50 when spec ≥ 10 KB. |
| G2 | Cross-task FAIL reasons of "doesn't match existing pattern / convention" drop to near-zero. | Manual review of FAIL ISSUES in `events.jsonl` across 3 plans of ≥ 10 tasks each. Threshold: <1 per 30 tasks. |
| G3 | Quality_trend mean does NOT decline through chain-resume boundaries. | `quality_score` mean of first 5 tasks after a chain handoff equals (within 0.05) mean of last 5 tasks before. |
| G4 | Token-based Resume Chain trigger fires before the legacy `compaction_points ≥ 2 AND completed ≥ 8` trigger for chain-resumed runs on 30+ task plans. | First chain handoff timestamp < timestamp where legacy trigger would have fired. |

## Non-goals

- No change to model selection (per v2.12).
- No change to retry budgets, parallel dispatch, hook contracts, learning-log lifecycle.
- No new sub-agent dispatches.
- No automatic conflict resolution between newly-made decisions and existing register entries — the register surfaces; the Combined Reviewer flags conflicts; the existing spec-edit branch handles resolution.
- No removal of the prose "drop prior context" instruction in T3 — kept as defense in depth alongside the mechanism.

## Architecture invariants preserved

- Schema version stays `"2"` (additive only).
- Per-plan `<active>` resolution rule (v2.13) unchanged. Spec manifest is run-level (one spec per plan; for chained plans, each plan_chain entry gets its own spec_manifest).
- Cost ledger ownership (v2.14 F2) unchanged. C3 reads from but does not mutate cost_ledger.
- Resume Chain handoff procedure (Phase 0 Step 0, Resume Chain procedure) unchanged in structure — only the trigger predicate changes.

## Feature C1 — Spec manifest (Tiered Context Injection)

### C1.1 New state.json fields

For single-plan: top-level. For multi-plan: under each `plan_chain[N]`. Resolves through `<active>` per v2.13 rule.

```json
"spec_manifest": {
  "spec_path": "<path>",
  "spec_total_chars": <int>,
  "sections": {
    "<section_id>": {
      "title": "<heading text>",
      "range": [<line_start>, <line_end>],
      "chars": <int>,
      "anchor": "<slug of heading for ToC>",
      "level": <int 1-6 — markdown heading level>
    }
  },
  "task_to_sections": {
    "task_0": ["<section_id>", "..."],
    "task_1": ["..."]
  },
  "fallback_policy": "full_spec_on_blocker" | "halt_on_blocker"
}
```

`<section_id>` format: `S<level>.<numeric path>`. Examples: `S1.1`, `S2.3.1`, `S1`. Generated by `scripts/build_spec_manifest.py` traversing markdown heading hierarchy.

`fallback_policy`:
- `"full_spec_on_blocker"` (default) — if Implementer ESCALATEs with SPEC_BLOCKER and the cause is *"missing context"*, the next dispatch inlines the full spec as fallback. Counts against `spec_clarifications`.
- `"halt_on_blocker"` — manifest is treated as authoritative; missing context halts task.

### C1.2 Manifest builder script

New: `scripts/build_spec_manifest.py`.

Input: spec file path. Output: JSON to stdout matching `spec_manifest.sections` shape (no task_to_sections — that's filled by Orchestrator).

Algorithm:
1. Parse markdown headings (`^#{1,6}\s+`).
2. Assign hierarchical IDs by depth-first traversal. Top-level (`#`) = `S1`, second = `S1.1` etc. Reset numeric path when heading level decreases.
3. For each heading, compute `range = (start_line, next_heading_line - 1 OR EOF)`.
4. Compute `chars` from file slice.
5. `anchor` = lowercase slug of heading text (alphanumerics + hyphens).
6. Emit sorted-by-section_id JSON.

Edge cases:
- Spec with no headings → emit single section `S0` covering entire file.
- Code blocks containing `#` characters at line start → ignore (detect fenced blocks).
- HTML comments wrapping headings → ignore wrapped section (do not emit).

### C1.3 task_to_sections mapping

Filled at Phase 0 Step 6 (dependency graph). For each task in the plan:

1. Parse task body for explicit `**Spec Refs:**` block:
   ```
   **Spec Refs:** S1.2, S3.1
   ```
   If present: use those section IDs verbatim. Validate each exists in spec_manifest.sections; unknown ID → Plan Reviewer BLOCKER.

2. Else: heuristic inference from `**Files:**` block:
   - For each file in Files, look up the spec section(s) whose `title` mentions any path component of the file (case-insensitive substring).
   - If 0 matches: fallback `["*"]` (full spec inlined for this task — same as today's behavior). Record as `fallback_used` in `spec_manifest.task_to_sections.<task>.fallback_used: true`.
   - If 1+ matches: use those section IDs.

3. Plan Reviewer (Phase 0 Step 6.5) gets new rubric items:
   - WARN if any task has `task_to_sections[<task>] == ["*"]` (heuristic fallback used).
   - BLOCKER if any task has `**Spec Refs:**` block with an unknown section ID.
   - WARN if a section is referenced by 0 tasks (likely dead spec content).

### C1.4 Prompt builder change

Phase 1 Step 1 Implementer prompt template — the `{relevant spec excerpt}` placeholder changes interpretation:

Before (v2.14): inline full spec or Orchestrator-curated excerpt of "relevant" sections (subjective).

After (v2.15):
```
For task_<N>:
  section_ids = state.spec_manifest.task_to_sections["task_<N>"]
  if "*" in section_ids:
    spec_text = full spec file contents
  else:
    spec_text = concatenated text of each section in section_ids
                (in spec_manifest order, with section heading prefixed)
  Replace {relevant spec excerpt} with spec_text
  Add prefix: "## Spec context (sections: {section_ids})\n\n"
```

If a sub-agent ESCALATEs SPEC_BLOCKER and the blocker mentions missing context (`cause` field contains "context" or "missing section" or "ambiguous reference"):
- If `fallback_policy == "full_spec_on_blocker"`: re-dispatch with FULL spec inlined; increment `spec_clarifications` (NOT review_retries — per v2.13 P15 rule).
- If `fallback_policy == "halt_on_blocker"`: standard ESCALATE handling.

### C1.5 Spec edit propagation

When the spec-edit branch fires (Phase 1 Step 2 SPEC_FAULT spec_contradicts/unclear), AFTER the spec edit commit succeeds:

1. Re-build spec_manifest sections (script re-run).
2. Recompute task_to_sections for incomplete downstream tasks whose sections overlap the edited line range.
3. Append a journal-style note to `state.spec_edits[*].manifest_recompute: true`.

Existing `[SPEC UPDATED]` prompt injection (P15) remains — now scoped to only the changed sections, not full spec.

### C1.6 Acceptance (C1)

A1. `scripts/build_spec_manifest.py` exists; smoke test (1-heading, 5-heading, 0-heading specs) produces valid JSON.
A2. After Phase 0 Step 6 on a 5-task plan with 10-heading spec: `state.spec_manifest.task_to_sections` populated for all 5 tasks; ≥ 3 tasks have specific section lists (not `["*"]`).
A3. Plan Reviewer dispatched against a plan with one task containing `**Spec Refs:** S99.99` (invalid ID) returns ISSUES_FOUND with severity BLOCKER, category `spec_manifest_invalid_ref`.
A4. Implementer dispatch for task with `task_to_sections=["S1.1"]` receives `{relevant spec excerpt}` that does NOT contain other sections' content (verifiable via prompt file dump).
A5. A/B vs v2.14: on a sample 20-task plan with 30 KB spec, median Implementer `input_tokens` post / pre ≤ 0.50.

## Feature C2 — Decisions register (cross-task injection)

### C2.1 New state.json fields

Per-plan (under `<active>`):

```json
"decisions_register": [
  {
    "task": "<task_id>",
    "decision": "<≤15-word text, copied from task_summaries.task_N.key_decision>",
    "files": ["<file>", "..."],
    "made_at": "<iso8601>",
    "supersedes": "<task_id>" | null
  }
]
```

`supersedes`: if a new decision explicitly overrides an earlier one (Combined Reviewer flags conflict; spec-edit branch resolves; Orchestrator records new entry with `supersedes` pointer). Superseded entries remain visible in the register but are prefixed `[SUPERSEDED by task_X]` when projected to prompts.

### C2.2 Accumulation point

Phase 1 Step 4 Agent Cleanup, AFTER task_summaries is written:

```
If task_summaries.<task_id>.key_decision is non-empty AND not literally "(none)":
  Append to <active>.decisions_register:
    {task, decision, files (from task_summaries), made_at: now, supersedes: null}
```

Atomic R-M-W of state.json. Failure of decision append → log warning, continue (decisions register is best-effort enrichment, not load-bearing).

### C2.3 Prompt injection — Implementer

Phase 1 Step 1 Implementer prompt template gains a new `{decisions_register}` placeholder, inserted near the top (above task description, below role declaration):

```
## Project decisions so far (do NOT re-decide; raise objections via Reviewer if any seem wrong):
- [task_3] JSON validation via ajv (perf 3x faster than zod) — src/validate.ts
- [task_8] API errors use Problem+JSON RFC 7807 — src/error.ts
- ...
```

Render order: by `made_at` ascending. Superseded entries: render with strikethrough prefix `~~[SUPERSEDED by task_X]~~`.

Empty register → omit the section entirely.

### C2.4 Prompt injection — Combined Reviewer

`references/reviewer-prompt.md` gains a new rubric item:

> **Decision consistency**: examine the diff. If this change introduces an approach that conflicts with an existing register entry (e.g., new code uses zod when register says ajv): flag as QUALITY_ISSUE with category `decision_conflict`. Reference the conflicting register entry's task ID.

This adds one new ISSUE_KEY category. Existing PASS/WARN/FAIL tier logic unchanged.

### C2.5 Decisions register projection to KNOWLEDGE-like file

Phase 2 Step 1 (Final Docs Updater) gains a new auto-generated artifact:

`<worktree>/.orchestrator/DECISIONS.md` — projection of `decisions_register` as human-readable markdown table. Re-rendered at every Phase Transition T3 and at Phase 2 Step 1. Sorted by made_at ascending. Superseded entries grouped at bottom.

This file is included in the archive tarball (F1).

### C2.6 Acceptance (C2)

A1. After 3 tasks complete with non-empty `key_decision`: `state.decisions_register` has 3 entries with correct task/decision/files/made_at.
A2. Implementer dispatch for task 5 receives prompt containing the `## Project decisions so far` block listing tasks 1, 2, 3 decisions.
A3. A diff intentionally introducing a conflicting library (test fixture: register says "use ajv", diff adds zod): Combined Reviewer output contains ISSUE_KEY with category `decision_conflict`.
A4. `<worktree>/.orchestrator/DECISIONS.md` exists after Phase 2 Step 1; contains all register entries in markdown table.
A5. Empty register → prompt template omits the section entirely (verifiable via prompt file dump).

## Feature C3 — Token-based Resume Chain trigger

### C3.1 Current trigger (v2.13)

> "Chain ONLY when both: `compaction_points ≥ 2` AND count of COMPLETE tasks `≥ 8`."

### C3.2 New trigger (v2.15)

```python
def should_chain(state):
    if state.budget_action == "off":
        # Without cost tracking, fall back to legacy heuristic.
        return state.compaction_points_reached >= 2 and complete_count(state) >= 8

    # Estimate input context window usage from cost ledger.
    # Sum NON-CACHED input tokens across all dispatches this session.
    session_input_tokens = (
        state.cost_ledger.totals.input_tokens
        - state.cost_ledger.totals.cached_read_tokens
    )

    # Threshold: 60% of effective context budget.
    # Sonnet/Opus: 200_000 tokens nominal; subtract 30k for prompt overhead → 170k effective.
    # 60% = 102_000 tokens.
    EFFECTIVE_BUDGET = 170_000
    THRESHOLD = int(EFFECTIVE_BUDGET * 0.60)

    if session_input_tokens >= THRESHOLD:
        return True

    # Also keep the legacy floor — chain if either condition met.
    return state.compaction_points_reached >= 2 and complete_count(state) >= 8
```

Rationale:
- Cache-read tokens excluded — they don't consume fresh input budget on retry sessions.
- Threshold 60% leaves 40% headroom for the final completion turn + tool-call overhead.
- Legacy floor preserved as belt-and-suspenders for `budget_action == "off"` and as backup for sub-200k-context runs.

### C3.3 Per-orchestrator-dispatch tracking

The Orchestrator (Opus) doesn't currently track its own context window directly. We approximate via cost_ledger:

Hot path: every time the Orchestrator's own conversation gains tokens (every Read, every Agent dispatch return), we cannot observe directly. Best available proxy: the sub-agent return text length + cost_ledger increment.

Approximation acceptable because:
- Trigger fires before *exhaustion*, not *exactly at* 60% — error margin is fine.
- Worst case (over-eager chain): extra subprocess spawn, ~5s + ~1k tokens cost. Negligible vs the alternative (context-saturated Orchestrator dropping signals).

### C3.4 Field additions

```json
"context_budget": {
  "effective_input_budget": 170000,
  "threshold_ratio": 0.60,
  "threshold_tokens": 102000,
  "last_evaluation_at": "<iso8601>",
  "last_evaluation_tokens": <int>
}
```

Top-level (run-level). Updated at each Phase Transition T3 (where trigger is evaluated).

`effective_input_budget` and `threshold_ratio` are configurable via new args:
- `context_budget=<tokens>` — override effective budget (e.g., 1000000 for upcoming 1M-context models).
- `context_threshold=<ratio>` — override threshold (0.0 to 0.95).

### C3.5 Trigger evaluation point

Same as today: Phase Transition T3 (post state-anchor, post context_health emit). NEW: also evaluate at end of Phase 1 Step 4 if `current_task` is a compaction_point AND token threshold crossed (so chain can fire even if just one compaction has occurred).

### C3.6 Telemetry — measure trigger lift

Every chain decision (chain OR no-chain) at Phase Transition T3 emits `context_health` event:

```json
{
  "event_type": "context_health",
  "issue_key": "chain_trigger_eval",
  "severity": "low",
  "summary": "Chain trigger eval: <chained|not_chained> | tokens=<N>/<threshold> | compactions=<N>/2 | completed=<N>/8",
  "context": {
    "trigger_decision": "chained" | "not_chained",
    "trigger_reason": "token_threshold" | "legacy_floor" | "none",
    "session_input_tokens": <int>,
    "threshold_tokens": <int>,
    "compactions_reached": <int>,
    "completed_count": <int>
  }
}
```

Enables post-hoc analysis: how often did token-trigger fire before legacy floor? Was 60% the right ratio?

### C3.7 Acceptance (C3)

A1. On a plan with cost_ledger.totals.input_tokens > 102000 after 2 compactions: chain fires.
A2. On a plan with cost_ledger.totals.input_tokens > 102000 after 1 compaction: chain fires (NEW behavior — would not have under v2.13).
A3. With `budget_action=off`: legacy trigger (compactions≥2 + completed≥8) is the only mechanism.
A4. With `context_budget=500000 context_threshold=0.75`: trigger evaluates at 375000 tokens.
A5. Every Phase Transition T3 emits one `chain_trigger_eval` event regardless of decision.
A6. A/B vs v2.14 on a 30-task plan with 15+ KB spec: v2.15 chains earlier (lower compaction_index of first handoff).

- Skip C1 entirely on resume if `state.spec_manifest` already populated AND spec file mtime ≤ manifest build time (cached).
